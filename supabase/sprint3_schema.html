¿Por qué tabla intermedia y no services[] en appointments?
Tu propuesta original tenía services como campo en appointments. Si fuera un array JSON o un text[], pierdes la capacidad de hacer queries útiles.
❌ SELECT * FROM appointments WHERE 'Baño' = ANY(services) -- frágil
❌ No puedes joinear con la tabla services para obtener el precio
❌ No puedes calcular "cuánto generó el servicio de baño este mes"
✓ SELECT SUM(price_at_time) FROM appointment_services WHERE service_id = 'uuid-baño'
✓ JOINs limpios, índices eficientes, queries analíticas posibles
¿Por qué price_at_time en appointment_services?
Si el precio de "Baño" cambia de $200 a $250, las citas anteriores deben conservar el precio que tenían. Si solo guardas el service_id y lees el precio de la tabla services, todas las citas históricas mostrarían $250 aunque se cobraron a $200.
✓ price_at_time = snapshot del precio en el momento de la cita
✓ Historial de pagos siempre correcto aunque cambien los precios
¿Por qué vet_schedules separado?
La disponibilidad del vet no pertenece a ninguna cita en particular — es información del vet que existe independientemente. Al tener tabla separada puedes calcular slots disponibles haciendo un query de "horario del vet MINUS citas ya agendadas".
